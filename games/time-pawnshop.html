Design Philosophy

为什么做这款游戏

核心理念 · 设计决策 · 落地实现

🌊

数据驱动叙事引擎

全部对话文本、角色关系、选择后果定义为 JSON 配置。策划可以完全独立地增加角色、扩展剧情、调整数值，无需触碰程序代码。程序只负责渲染引擎和数据解析器。

🔄

双层存档架构（MetaSave）

第一层：当前周目的剧情节点、对话进度、选择状态。第二层：跨周目持久化（全局真相解锁项、隐藏结局开放状态）。玩家每次重新开始都带着上一轮的认知前进，而非从零开始。

🎭

蝴蝶效应因果网

每个选择产生「命运涟漪」，涟漪在后台积累，跨越时间线影响其他角色的命运。玩家无法预知自己的选择会如何连锁反应——但每次回溯周目，都会发现命运早已暗流涌动。

📐

四层叙事结构

「现在」经营层 →「过去」回忆层 →「真相」元叙事层 →「终局」终章层。玩家在第四层才能拼凑出游戏的完整真相，理解为什么「时间当铺」本身也是故事的一部分。

游戏数据模型（data.js）

data.js · 核心数据结构
数据类型 用途说明 设计亮点
CHARACTERS 角色基础信息（名称、描述、身份） 支持多身份标签（NPC / 涟漪携带者 / 终局角色）
NODES 剧情节点（节点ID、显示内容、角色、出场时间线） 节点可配置「前置涟漪条件」，条件满足才出现
CHOICES 玩家选择（选择文本、触发涟漪、后续节点） 选择结果以涟漪ID记录，而非直接写死后续节点
RIPPLES 命运涟漪定义（涟漪ID、名称、传播方向、影响规则） 涟漪有「正向传播」和「跨时间线传递」两种模式
TIMELINES 时间线定义（1962 / 1997 / 2024 / ????） 跨时间线数据同步，涟漪可以穿越时间线
TRUTHS / SECRETS 真相碎片 / 隐藏内容（MetaSave 层） 跨周目累积，解锁真结局的前置条件

引擎架构（engine.js / game.js）

⚙️

Engine：渲染引擎

管理 Canvas 渲染循环、像素风画面绘制、对话框展示、角色立绘、背景切换。纯渲染逻辑，不含游戏状态。

🎮

Game：游戏逻辑层

管理游戏状态机（开始菜单→剧情节点→选择→涟漪计算→下一节点→结局判定）、存档读写（localStorage + MetaSave 合并）。

💾

双层存档系统

localStorage 存储当前周目状态；MetaSave 存储跨周目解锁数据。二者在游戏启动时合并，确保真结局的完整性。

📡

涟漪传播算法

每次选择后，Game 层查询 RIPPLES 表，计算涟漪的目标时间线和目标角色，更新全局涟漪状态。下一节点加载时，节点前置条件被实时计算。

游戏概述

类型：像素风叙事解谜 · HTML5 Canvas
平台：Web（浏览器直接运行）
核心循环：接待客人 → 进入记忆 → 做出选择 → 产生涟漪 → 观察后果
目标用户：喜欢叙事解谜、重视剧情深度的独立游戏玩家

🗓️

开发周期

2024 年完成，独立策划 + 原生 JavaScript 开发
游戏源码：game.js (2123行) + engine.js (899行) + data.js (2567行)
无第三方游戏引擎依赖，纯原生 Canvas 实现

🎯

核心设计目标

① 每个选择的后果不可预测，但逻辑自洽
② 玩家可以清晰地感知到「选择有代价」
③ 多周目体验不重复，新周目有新发现
④ 真相碎片分散在不同周目，鼓励玩家反复探索

🔢

数值与进度体系

• 命运涟漪值：全局累计，影响所有节点的可见性
• 角色亲密度：以涟漪为载体，跨时间线传递
• 真相碎片进度：MetaSave 层，影响结局走向
• 结局判定：涟漪值 × 真相碎片数 × 特殊标记

「时间当铺的核心不是'解谜'，而是'理解'。玩家每一次进入记忆，都是在用现在的认知重新审视过去的选择。」
