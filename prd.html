一、背景与目标

1.1 业务背景

00:00Space Club 是一家位于美国纽约的线下娱乐场所，主营业务包括：

  • 场地租赁（DJ台、舞池、VIP包厢）
  • 酒水销售（现场调酒、瓶装酒、软饮）
  • 活动策划（主题派对、私人定制活动）

当前痛点：

  • 酒水销售佣金依赖人工核算，每周耗费店长3-4小时
  • 提成规则复杂（分级提成 + 品类差异 + 时段加成），人工计算易出错
  • 延迟结算影响员工积极性，导致销售动力不足
  • 缺乏数据沉淀，无法分析高利润品类/时段/员工

1.2 项目目标

目标类型 具体指标 验证方式
效率目标 佣金核算时间从3小时/周降至5分钟/周 系统操作日志统计
准确目标 核算错误率从5%降至0.1% 财务对账差异统计
体验目标 员工可实时查看预估佣金 员工满意度调研（NPS≥40）
数据目标 建立酒水销售数据看板 看板使用频率统计

1.3 目标用户

用户角色 核心需求 使用场景 痛点
店长（Admin） 高效准确核算、规则灵活配置 每周结算日 人工核算耗时、规则复杂易错
调酒师/服务员（Staff） 实时查看佣金、理解提成规则 每日下班前 佣金不透明、延迟发放
财务（Finance） 数据可追溯、报表导出 月度审计 数据分散、对账困难

二、用户故事

2.1 店长视角

US-001
规则配置
作为店长，我希望在系统中配置不同酒水品类的提成比例，以便佣金规则可灵活调整。
Acceptance Criteria
  • 支持按品类（啤酒/烈酒/鸡尾酒/软饮）设置基础提成比例
  • 支持按员工等级（初级/中级/高级）设置系数加成
  • 支持按时段（平日/周末/节假日）设置加成系数
  • 规则修改后，历史订单佣金按原规则计算，新订单按新规则计算
US-002
批量核算
作为店长，我希望一键核算本周所有员工的酒水销售佣金，以便快速完成结算。
Acceptance Criteria
  • 支持选择时间范围（本周/本月/自定义）
  • 自动读取POS系统销售数据
  • 按规则自动计算每位员工佣金
  • 生成可导出的结算报表（PDF/Excel）
  • 异常数据（如退货、折扣订单）高亮提示
US-003
数据看板
作为店长，我希望查看酒水销售的数据看板，以便优化进货和排班策略。
Acceptance Criteria
  • 展示Top10高利润品类
  • 展示各时段销售热力图
  • 展示员工销售排名
  • 支持按周/月/季度筛选

2.2 员工视角

US-004
实时佣金查询
作为调酒师，我希望在手机上实时查看今日预估佣金，以便了解工作成果。
Acceptance Criteria
  • 移动端适配（微信小程序）
  • 展示今日/本周/本月累计佣金
  • 展示每笔订单的佣金明细（订单号、品类、金额、提成比例、佣金）
  • 佣金数据T+1更新（当日数据次日可见）
US-005
规则透明
作为服务员，我希望清楚了解提成规则，以便知道如何提升收入。
Acceptance Criteria
  • 展示当前生效的提成规则（品类比例、等级系数、时段加成）
  • 用示例说明规则计算逻辑
  • 规则变更时推送通知

2.3 财务视角

US-006
审计追溯
作为财务人员，我希望查看每笔佣金的计算明细和规则版本，以便审计核查。
Acceptance Criteria
  • 每笔佣金记录关联：员工、订单、规则版本、计算时间
  • 支持按员工/时间/订单号检索
  • 支持导出审计报表
  • 规则变更历史留痕

3.3 核心功能详细设计

3.3.1 佣金计算引擎

单笔佣金 = 订单金额 × 品类基础比例 × 员工等级系数 × 时段加成系数 总佣金 = Σ(单笔佣金) - 退货扣减 - 折扣分摊

规则优先级（冲突时以高优先级为准）：

  1. 特殊活动规则（临时促销）
  2. 节假日时段规则
  3. 周末时段规则
  4. 平日时段规则
  5. 品类默认规则

异常处理：

  • 退货订单：佣金自动扣减，历史已发放佣金标记为"待追回"
  • 折扣订单：按实际成交金额计算佣金
  • 多人协作订单：支持分佣比例配置（如调酒师70% + 服务员30%）

3.3.2 订单同步机制

同步方式 适用场景 技术方案 优先级
API对接 POS系统支持开放接口 定时拉取（每15分钟） P0
CSV导入 POS系统不支持API 模板下载 → 填写 → 上传 → 数据校验 P0
手动录入 特殊订单/补录 表单录入 + 审批流 P1

七、技术可行性评估

7.1 技术方案选型

技术环节 推荐方案 选型理由 风险
前端（管理端） React + Ant Design 组件丰富、开发效率高 低
前端（小程序） 微信小程序原生 无需额外适配成本 低
后端 Node.js / Express 快速开发、JSON处理友好 低
数据库 PostgreSQL 支持复杂查询、事务安全 低
订单同步 定时任务（node-cron） 实现简单、可扩展为消息队列 中
报表导出 PDFKit + xlsx 纯Node.js实现，无需额外服务 低
部署 Docker + AWS/阿里云 标准化部署、易维护 低

7.2 技术风险与应对

风险点 影响 应对方案
POS系统无开放API 订单同步依赖手动CSV导入，效率低 MVP阶段接受CSV导入，同步推进API对接谈判
佣金规则频繁变更 历史数据追溯复杂 规则版本化设计，每笔佣金记录关联规则版本ID
并发计算性能瓶颈 高峰期核算卡顿 计算任务异步化（消息队列），前端展示进度条
数据安全（员工隐私） 佣金数据敏感 字段级加密 + 操作日志审计 + 最小权限原则

八、数据验证方案

8.1 A/B测试方案

测试假设：员工实时查看佣金可提升销售积极性，进而提升酒水销售额。

维度 方案
实验组 开通小程序实时佣金查询功能
对照组 维持原有周结算告知方式
核心指标 人均酒水销售额（周环比）
辅助指标 员工满意度、系统使用率
实验周期 4周
样本量 全部员工（约15人），无需分组
显著性标准 人均销售额提升≥10%，p<0.05

8.2 灰度发布方案

阶段 范围 验证目标 周期
内测 店长 + 1名调酒师 核心流程跑通、无阻塞BUG 3天
小范围 50%员工 真实场景验证、收集反馈 1周
全量 全部员工 全面使用、数据监控 持续
